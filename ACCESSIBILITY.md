# Accessibility Statement and Review Workflow

## Commitment

AI Security Manuals is intended to be usable by people with different abilities, technologies, learning preferences, and levels of professional experience. Accessibility improvement is an ongoing responsibility rather than a one-time certification.

Publication does not represent a claim that every file formally conforms to WCAG 2.2, PDF/UA, or another accessibility standard.

## Current Formats

- PDF editions support convenient reading and distribution.
- DOCX editions support editing, notes, training, and assistive-technology workflows.

When one format presents a barrier, readers are encouraged to try the other format and report the problem.

## Review Checklist

Each manual should be checked for:

1. Correct document language
2. Descriptive title
3. Logical heading hierarchy
4. Logical reading order
5. Meaningful link text
6. Alternative text for meaningful images
7. Decorative images marked appropriately
8. Sufficient color contrast
9. Tables with proper header identification
10. Lists created as real lists
11. Keyboard-accessible navigation
12. Usability at 200 percent zoom
13. Clear language and explained abbreviations
14. Tagged PDF structure when PDF is provided
15. Bookmarks for longer PDF documents

## Testing Levels

The project uses three review labels:

- **Not yet evaluated** — no accessibility review recorded
- **Structural and automated checks completed** — document structure and automated checks reviewed
- **Manual accessibility review completed** — keyboard, reading order, zoom, and representative assistive-technology checks completed

Automated checks do not prove full accessibility.

## Correction Workflow

1. Record the manual, language, version, format, page, and barrier.
2. Describe the expected accessible behavior.
3. Avoid including medical information or other sensitive personal details.
4. Correct the source document first when possible.
5. Regenerate derivative formats.
6. Recheck the affected requirement.
7. Record the correction in the changelog.

## Reporting a Barrier

Use GitHub Issues and include:

- Manual and language
- PDF or DOCX
- Version
- Page or section
- Assistive technology or viewing method, when relevant and comfortable to disclose
- Description of the barrier
- Suggested correction, if known

Disability or medical disclosure is never required.

## Priority

Issues preventing access to essential content, navigation, warnings, or download links receive priority over cosmetic improvements.
